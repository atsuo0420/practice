# practice ディレクトリのルール

## 目的
ここは Claude Code の練習用ディレクトリ。将来的に企業に売れるレベルのシステムを作れるようになるために、いきなり本番・納品レベルを目指すのではなく、まず自分自身が欲しい・使いたいシステムを自由に作って練習する場所。

## 構成
- 1プロジェクト = 1サブフォルダ。`001-xxx`, `002-xxx`, ... と連番＋内容がわかる名前をつける。
- 各プロジェクトフォルダには簡単な `README.md` を置く（目的・使い方・現状のステータスを数行で）。チャットはプロジェクトごとに分けて進める運用のため、新しいチャットを開いたときにファイルを読むだけで状況を把握できるようにする。
- 現在のプロジェクト:
  - `001-ai-daily-logger`: 自由入力の日報テキストを分類してNotion向けMarkdownに整形するツール
  - `002-morning-brief`: 通勤中に読む個人用ニュースダイジェント（NewsAPI利用）

## 進め方の方針
- 各プロジェクトは小さく・実用的に。過剰設計や将来を見越した抽象化はしない。
- 技術スタックは基本 vanilla HTML/CSS/JS など、シンプルなものから。案件ごとに違う技術に触れて経験の幅を広げるのもOK。
- 基本は新しい番号のプロジェクトに軸足を移しつつ、進行中でも過去のプロジェクトに思いついた機能を随時追加してよい。「完成→次へ」の一直線ではなく、複数プロジェクトを並行して育てていくスタイル。
- あるプロジェクトが完全に自分の思い通りに動く状態になったら、そこで完成とみなす。完成したプロジェクトはそれ以上いじらず、対応するチャットも役目を終えて閉じてよい。
- ユーザーはプログラミング初心者。ターミナル・git・VSCodeなどの基本操作は都度、前提知識ゼロから説明する。

## 秘密情報の扱い
- APIキーなどの秘密情報はファイルやgitにコミットせず、ブラウザのUI入力＋`localStorage`保存などに留め、リポジトリはいつでも安全に共有・コミットできる状態を保つ。

## git運用
- practiceディレクトリ全体を1つのgitリポジトリとして管理する。
- commitは毎回確認を取らず、区切りがいいタイミング（機能がひとつ動くようになった、プロジェクトが一段落したなど）でClaudeが自動でこまめに行ってよい（ユーザーが2026-07-16に許可）。1つの巨大なcommitにまとめない。
  - ユーザーが「commitは自分から指示する形に変えたい」と言ったら、このcommitの自動実行はやめて、都度確認する運用に戻す。
- push・force-pushなどリモートに影響する操作や、reset --hardなど元に戻せない操作は、この自動commitの許可があっても必ず都度ユーザーに確認してから行う。
